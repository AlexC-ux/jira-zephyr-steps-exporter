## 📋 Описание

Скрипт для автоматической выгрузки шагов тестовых кейсов из Jira с использованием плагина Zephyr for Jira. Результат сохраняется в файл формата ODS (OpenDocument Spreadsheet), который совместим с Microsoft Excel, LibreOffice Calc, Google Sheets и другими табличными редакторами.

**Основное назначение:**
- Экспорт тестовых шагов по заданному JQL-запросу
- Генерация отчетов по тестовым сценариям для документации
- Подготовка данных для аудита качества ПО
- Автоматизация процесса документирования тестовых кейсов
- Анализ покрытия тестами бизнес-логики
- Экспорт данных для миграции между системами управления тестами

**Используемые технологии:**
- **Jira REST API** — получение задач и метаданных из Jira
- **Zephyr for Jira Server API** — получение тестовых результатов и шагов
- **JSDOM** — парсинг HTML-содержимого полей Jira
- **SheetJS (xlsx)** — генерация таблиц формата ODS/CSV
- **JSONPath** — извлечение данных из кастомных полей

#### 💡 Советы по JQL-запросам

Примеры полезных JQL-запросов для поиска задач с тестовыми кейсами:

| Запрос | Описание |
|--------|----------|
| `project = PROJ` | Все задачи проекта |
| `project = PROJ AND status = Done` | Завершенные задачи проекта |
| `project = PROJ AND status != Done` | Незавершенные задачи |
| `project = PROJ AND assignee = currentUser()` | Задачи текущего пользователя |
| `project = PROJ AND assignee = "ivan.ivanov"` | Задачи конкретного пользователя |
| `project = PROJ AND updated >= -30d` | Обновленные за последние 30 дней |
| `project = PROJ AND created >= -7d` | Созданные за последнюю неделю |
| `key = PROJ-123` | Конкретная задача по ключу |
| `issue in hasTestCoverage()` | Все задачи, имеющие сценарий Zephyr |
| `issue in testCasesWithSteps()` | Задачи с заполненными тест-шагами |
| `project = PROJ AND priority = High` | Задачи высокого приоритета |
| `project = PROJ AND labels = "regression"` | Задачи с меткой regression |
| `project in (PROJ1, PROJ2, PROJ3)` | Несколько проектов |
| `project = PROJ AND Sprint in openSprints()` | Задачи из открытых спринтов |
| `project = PROJ AND fixVersion = "v2.0"` | Задачи для конкретного релиза |

### Кастомные поля Jira

Для добавления кастомных (пользовательских) полей задачи в выгрузку используется переменная окружения `custom_jira_fields_expr`. Это позволяет извлечь дополнительную информацию из полей задач Jira и включить её в выходной файл для расширенного анализа.

#### Синтаксис выражений

В файле `.env` задайте выражение(я) в формате JSONPath, разделяя их запятыми:

```env
# Одно кастомное поле
custom_jira_fields_expr="customfield_200[*].value"

# Несколько кастомных полей (через запятую)
custom_jira_fields_expr="customfield_1300[*].value,customfield_10100[*].value,customfield_200[*].value"
```

#### Как работает извлечение

- `customfield_200[*].value` — извлекает значение из кастомного поля с ID 200
- `[*]` — означает "все элементы массива" (для multi-select полей)
- `.value` — указывает, что нужно взять значение поля (если оно структурированное)
- Несколько выражений через запятую добавляют несколько столбцов в выходной файл

#### Типичные примеры использования

| Выражение | Описание поля |
|-----------|---------------|
| `customfield_200[*].value` | Изменить статус (Custom Field ID: 200) |
| `customfield_1300[*].value` | Приоритет теста (Custom Field ID: 1300) |
| `customfield_10100[*].value` | Сложность теста (Custom Field ID: 10100) |
| `customfield_10200[*].value` | Тип теста (unit/integration/e2e) |
| `customfield_10300[*].value` | Критичность теста |
| `customfield_10400[*].value` | Время выполнения теста |
| `customfield_10500[*].value` | Автор теста |
| `customfield_10600[*].value` | Дата последнего изменения |

**Как найти ID кастомного поля:**
1. Откройте задачу Jira в браузере
2. Нажмите F12 (Developer Tools)
3. Откройте вкладку Elements
4. Наведите на нужное поле → нажмите правую кнопку мыши → "Inspect"
5. Найдите в HTML атрибute `data-field-id` или посмотрите ID в URL при редактировании

#### Пример настройки `.env`

```env
jira_url="https://jira.company.ru"
jira_username="qa.team@company.ru"
jira_password="your-api-token"
jira_jql="project = TEST AND status = Done"
custom_jira_fields_expr="customfield_1300[*].value,customfield_10100[*].value"
outpit_file_name="test_steps_with_metadata"
```

Результат: в выходном файле появятся дополнительные столбцы с именами `customfield_1300[*].value` и `customfield_10100[*].value`, содержащими значения соответствующих кастомных полей.

## 📊 Выходной формат

### Структура файла ODS

Файл содержит один лист `Test Steps` со следующими колонками:

| Колонка                  | Описание | Пример |
|--------------------------|----------|--------|
| **Task Key**             | Ключ задачи Jira | `PROJ-123` |
| **Test Result Key**      | Ключ тестового результата | `TR-456` |
| **Step Index**           | Номер шага в тесте | `1`, `2`, `3` |
| **Action (Description)** | Действие (описание шага теста) | `Открыть страницу входа` |
| **Expected Result**      | Ожидаемый результат шага | `Отображается форма входа` |
| **[custom_field]**       | Значение кастомного поля (из `custom_jira_fields_expr`) | `High` |

### Особенности форматирования

- ✅ **HTML парсинг** — все HTML-теги автоматически удаляются из описаний
- ✅ **Объединение ячеек** — значения Task Key и Test Result Key объединяются по группам для удобства чтения
- ✅ **Сортировка** — данные отсортированы по Task Key → Test Result Key → Step Index
- ✅ **Нормализация текста** — лишние пробелы и переносы строк удаляются
- ✅ **Поддержка Unicode** — кириллица и другие символы отображаются корректно

| Task Key | Test Result Key | Step Index | Action                 | Expected Result                |

## 🔧 Технические детали

### Используемые API

#### Jira REST API

```
GET /rest/api/2/search?jql={jql}&startAt={start}&maxResults={size}&expand=changelog
```

Параметры запроса:
- `jql` — JQL-запрос для поиска задач
- `startAt` — индекс первой возвращаемой задачи (для пагинации)
- `maxResults` — максимальное количество задач в ответе (до 1000)
- `expand` — дополнительные данные (changelog для истории изменений)

#### Zephyr for Jira Server API

```
# Получение связанных тестовых результатов для задачи
GET /rest/tests/1.0/issue/{issueId}/tracelinks?maxResults=50&startAt=0

# Получение шагов тестового результата
GET /rest/tests/1.0/testresult/{testResultKey}?fields={fields}
```

### Основные функции скрипта

| Функция | Описание |
|---------|----------|
| `getIssuesByJql(jql, startAt)` | Получение задач по JQL с автоматической пагинацией |
| `getTestResultsByIssue(issueId)` | Получение тестовых результатов для задачи через traceLinks |
| `getTestScriptResults(testResultKey)` | Получение шагов тестового результата |
| `formatFieldValue(value)` | Очистка HTML и нормализация текста (удаление тегов, пробелов) |
| `calculateMerges(data)` | Вычисление диапазонов для объединения ячеек в ODS |
| `generateODS(data)` | Генерация ODS файла с использованием SheetJS |

### Обработка ошибок

| Тип ошибки | Поведение |
|------------|-----------|
| **HTTP ошибки Jira API** (401, 403, 404, 500) | Выбрасывается исключение с описанием, скрипт завершается |
| **Ошибки при получении тестовых данных** | Логируется предупреждение, обработка продолжается |
| **Ошибки записи файла** | Скрипт завершается с кодом 1 и сообщением об ошибке |
| **JSON parse ошибки** | Логируется предупреждение, строка пропускается |

2. Выберите **Security** → **Create and manage API tokens**
3. Нажмите **Create API token**
4. Скопируйте токен и используйте вместо пароля

## 🔍 Решение типичных проблем

### ❌ Ошибка авторизации (401 Unauthorized)

**Симптом:** `Error: Request failed with status code 401`

**Причины и решения:**

| Причина | Решение |
|---------|---------|
| Неверный пароль | Проверьте правильность логина и пароля |
| API token не активен | Создайте новый API token в настройках профиля |
| Пароль устарел | Измените пароль в Jira и обновите `.env` |
| Проблемы с Two-Factor Authentication | Используйте API token вместо пароля |

### ❌ Ошибка доступа (403 Forbidden)

**Симптом:** `Error: Request failed with status code 403`

**Решения:**
- Проверьте права доступа к задачам по JQL
- Убедитесь, что у вас есть право "Browse Projects"
- Попробуйте упростить JQL-запрос для тестирования

### ❌ Нет данных в результате

**Симптом:** `Обработано задач: 0` или `Найдено testResult: 0`

**Проверьте:**
1. Правильность JQL запроса в интерфейсе Jira
2. Наличие привязанных тестовых кейсов к задачам
3. Доступность Zephyr API для вашего пользователя
4. Логи на наличие предупреждений

### ❌ Ошибка EBUSY (resource busy or locked)

**Симптом:** `Error: EBUSY: resource busy or locked, open '...\output\test_steps_export.ods'`

**Причины:**
- Выходной файл открыт в другом приложении (Excel, LibreOffice, Google Sheets)
- Антивирус или системная индексация временно блокируют файл
- Предыдущий запуск скрипта был прерван, но файл остался заблокированным

**Решение:**
1. Закройте файл, если он открыт в любом приложении
2. Перезапустите скрипт с другим именем выходного файла (измените `outpit_file_name`)
3. Дождитесь завершения операций антивируса (можно временно отключить)
4. Перезагрузите компьютер для сброса всех блокировок

### ❌ Превышение лимитов API

**Симптом:** Множество ошибок 429 (Too Many Requests)

**Решения:**
- Уменьшите значение `jira_task_load_chunk` до 50 или 25
- Добавьте задержки между запросами (в коде)
- Запускайте скрипт в нерабочее время

## 📚 Применение / Use Cases

### 1. Автоматизация документирования тестовых сценариев

**Сценарий:** QA команда ежемесячно экспортирует все тестовые кейсы для обновления документации.

**Настройка:**
```env
jira_jql="project = QA AND labels = 'test-case' AND status = Done"
outpit_file_name="test_documentation_2024"
```

### 2. Аудит покрытия тестами

**Сценарий:** Менеджер продукта проверяет покрытие тестами требований перед релизом.

**Настройка:**
```env
jira_jql="project = PROJ AND fixVersion = v2.0 AND issue in testCasesWithSteps()"
outpit_file_name="coverage_report_v2.0"
```

### 3. Миграция тестов между системами

**Сценарий:** Команда переходит с одной системы управления тестами в другую.

**Процесс:**
1. Экспортируйте тесты из текущей системы в ODS
2. Преобразуйте формат с помощью скрипта
3. Импортируйте в новую систему

### 4. Интеграция в CI/CD

**Сценарий:** Автоматический экспорт тестов при каждом пайплайне.

**Пример Jenkins pipeline:**
```groovy
pipeline {
    stages {
        stage('Export Tests') {
            steps {
                sh 'npm ci'
                sh 'cp .env.production .env'
                sh 'npm start'
            }
        }
    }
}
```

### 5. Анализ качества тестов

**Сценарий:** Тимлид анализирует количество шагов в тестах для оптимизации.

**Анализ:**
- Среднее количество шагов на тест
- Тесты с >20 шагов (возможно слишком сложные)
- Тесты с <3 шагов (возможно недостаточно детальные)
